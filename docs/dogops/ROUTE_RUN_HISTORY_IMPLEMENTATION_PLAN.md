# Route Run History Implementation Plan

## Scope

DogOps can currently execute one selected authored route through the dashboard `Run Route` button and persist the latest execution state in `route_execution.json`. This plan covers the next product step: turn route execution into a first-class run timeline with durable history, route action steps, evidence capture, and dashboard playback.

Primary source of truth remains `SPEC.md`. The implementation must use DimOS navigation, Go2, AprilTag, and existing DogOps store patterns first. It must stay demoable offline, and real-Go2 claims must be backed by real runtime evidence.

## Current State

- Dashboard route start is a single `Run Route` button after a route is selected.
- `Run Route` calls `/api/map/routes/follow`, which calls the DogOps MCP `follow_route` skill.
- `DogOpsRouteExecutor` sends selected route waypoints and records the latest execution to:

```text
.dogops/runs/<run_id>/route_execution.json
```

- `route_execution.json` stores route state, active waypoint, counts, last error, and events such as `queued`, `sent`, `reached`, `timeout`, `failed`, and `stopped`.
- Completed route events append summarized navigation evidence into `nav_events.jsonl`, `state.json`, `report.json`, and `report.md`.
- Observations, incidents, work orders, and nav events already use JSON/JSONL files under the run directory.
- There is no timestamp-indexed history of route attempts. Re-running a route overwrites `route_execution.json`.
- Route execution does not yet model action steps such as take picture, scan AprilTag/QR, inspect package, read gauge, or verify work order.

## Goal

Add durable route-run history and action-level state so DogOps can answer:

```text
Which route run is active?
Where is the dog now?
What waypoint/action is it doing?
What did it already do?
What does it plan to do next?
What evidence did it collect?
What happened across previous route runs?
```

The dashboard should show both current execution and historical route runs, with enough detail to support the demo, debugging, and post-run reports.

## Database Recommendation

Use a global local SQLite database for route-run history across all DogOps runs, while keeping per-run JSON/JSONL files for export compatibility and simple offline inspection.

Reasons:

- Route history is relational: route runs have many events, waypoints, actions, observations, evidence files, and status transitions.
- The dashboard will need fast filtered queries: latest run, active run, failed runs, runs by route ID, events for a run, evidence for a waypoint, and time-ordered playback.
- SQLite is local, serverless, deterministic, easy to back up with the run directory, and does not require cloud secrets.
- It avoids repeatedly scanning many JSONL files as run history grows across multiple demo and hardware runs.
- It can be introduced without adding a new service or external dependency.

Recommended persisted files:

```text
.dogops/
  dogops.sqlite                    # global route-run/history database
.dogops/runs/<run_id>/
  route_execution.json              # latest active/last execution compatibility view
  route_runs/
    <route_run_id>/
      route_run.json                # export/snapshot for this attempt
      events.jsonl                  # append-only event export
      evidence/
        ...
```

SQLite should be the query source for the dashboard across all local runs. JSON/JSONL should be generated or mirrored inside each run directory for portability, reports, and easy debugging.

Do not introduce Postgres or a network database for the base demo. A hosted database can be a later sync/export target if multi-operator history becomes important.

## Product Decisions

- Route run history is global across all local DogOps runs, not only inside the active `.dogops/runs/<run_id>` directory.
- Every `Run Route` click creates a new historical route run with a unique `route_run_id`.
- Route actions live in both mission YAML and authored routes:
  - mission YAML defines reusable/default expected actions for named routes or checkpoints;
  - authored routes can snapshot and override actions for a specific operational run;
  - route execution always uses the immutable action snapshot captured at start.
- DogOps should support both QR codes and AprilTags, and the action system must remain extensible for future dog-run actions.
- Until real Go2 camera frames are wired, `capture_image` should attach a placeholder demo image that is clearly generated by the DogOps demo path, not manual/operator-attached evidence.
- The dashboard timeline should include all relevant events: route execution, waypoint movement, route actions, observations, evidence, incidents, work orders, and verification events.
- Local history is retained indefinitely for now. Do not add automatic pruning in the first implementation.

### RouteAction

Route waypoints should be able to include planned actions. Start with a minimal schema:

```text
action_id: str
waypoint_id: str
kind: scan_tags | scan_qr | capture_image | inspect_asset | verify_work_order | wait | operator_prompt
required: bool
timeout_s: float
args_json: object
```

Examples:

```text
INBOUND_DOCK:
  - scan_tags expected=[20, 101, 102, 103]
  - scan_qr expected_prefix=PKG-

COOLING_1:
  - capture_image target=COOLING_1
  - inspect_asset asset_id=COOLING_1

QA_HOLD:
  - verify_work_order work_order_id=WO-001
```

Mission YAML should hold canonical default actions for the demo route. Authored route state should store action overrides and the selected route snapshot used for execution. This keeps the demo repeatable while still allowing dashboard-authored routes to add, remove, or tune actions without immediately rewriting canonical mission YAML.

## SQLite Schema

Initial tables:

```sql
CREATE TABLE route_runs (
  route_run_id TEXT PRIMARY KEY,
  dogops_run_id TEXT NOT NULL,
  run_dir TEXT NOT NULL,
  route_id TEXT NOT NULL,
  route_label TEXT,
  mission_id TEXT,
  state TEXT NOT NULL,
  started_at REAL NOT NULL,
  completed_at REAL,
  operator_id TEXT,
  dry_run INTEGER NOT NULL DEFAULT 0,
  transport TEXT NOT NULL,
  frame TEXT NOT NULL,
  selected_route_snapshot_json TEXT NOT NULL,
  active_waypoint_id TEXT,
  active_action_id TEXT,
  waypoints_total INTEGER NOT NULL DEFAULT 0,
  waypoints_reached INTEGER NOT NULL DEFAULT 0,
  actions_total INTEGER NOT NULL DEFAULT 0,
  actions_completed INTEGER NOT NULL DEFAULT 0,
  last_error TEXT,
  summary TEXT NOT NULL DEFAULT ''
);

CREATE TABLE route_run_events (
  event_id TEXT PRIMARY KEY,
  route_run_id TEXT NOT NULL,
  ts REAL NOT NULL,
  sequence INTEGER NOT NULL,
  kind TEXT NOT NULL,
  state TEXT NOT NULL,
  waypoint_id TEXT,
  action_id TEXT,
  target_id TEXT,
  x REAL,
  y REAL,
  error_m REAL,
  retries INTEGER NOT NULL DEFAULT 0,
  guided INTEGER NOT NULL DEFAULT 0,
  payload_json TEXT NOT NULL DEFAULT '{}',
  note TEXT NOT NULL DEFAULT '',
  FOREIGN KEY(route_run_id) REFERENCES route_runs(route_run_id)
);

CREATE TABLE route_run_evidence (
  evidence_id TEXT PRIMARY KEY,
  route_run_id TEXT NOT NULL,
  event_id TEXT,
  observation_id TEXT,
  kind TEXT NOT NULL,
  path TEXT,
  uri TEXT,
  sha256 TEXT,
  mime_type TEXT,
  metadata_json TEXT NOT NULL DEFAULT '{}',
  created_at REAL NOT NULL,
  FOREIGN KEY(route_run_id) REFERENCES route_runs(route_run_id),
  FOREIGN KEY(event_id) REFERENCES route_run_events(event_id)
);

CREATE INDEX idx_route_runs_started_at ON route_runs(started_at DESC);
CREATE INDEX idx_route_runs_dogops_run_id ON route_runs(dogops_run_id, started_at DESC);
CREATE INDEX idx_route_runs_route_id ON route_runs(route_id, started_at DESC);
CREATE INDEX idx_route_run_events_run_sequence ON route_run_events(route_run_id, sequence);
CREATE INDEX idx_route_run_evidence_run ON route_run_evidence(route_run_id, created_at);
```

## Execution Flow

1. Operator selects a route.
2. Operator clicks `Run Route`.
3. Server creates a new `route_run_id` and immutable route snapshot.
4. Dashboard immediately shows the run as `running`.
5. For each waypoint:
   - write `waypoint started`;
   - publish DimOS navigation goal;
   - write `goal sent`;
   - poll live odom/path/target;
   - write `reached`, `timeout`, `failed`, or `stopped`.
6. For each planned action at the waypoint:
   - write `action started`;
   - call the matching action handler;
   - persist observations/evidence;
   - write `action completed` or failure.
7. Append compatible DogOps `NavEvent` and `Observation` records.
8. Update current route execution view and dashboard status.
9. On finish, write `route_run.json`, `events.jsonl`, update report summaries, and keep the SQLite record queryable.

## Route Action Handlers

Start with deterministic/local handlers:

| Action kind | First implementation |
|---|---|
| `scan_tags` | Record deterministic demo detections from authored or mission-derived expected tag IDs, with evidence metadata. |
| `scan_qr` | Record deterministic demo detections from authored or mission-derived expected QR payloads, with evidence metadata. |
| `capture_image` | Copy a configured Go2 camera image path into evidence when supplied, otherwise save a deterministic placeholder demo image that looks like dog-captured evidence. |
| `inspect_asset` | Reuse existing `check_clearance`, `read_gauge`, or deterministic fact checks. |
| `verify_work_order` | Reuse `verify_work_order` and attach current route/evidence context. |
| `wait` | Deterministic timer with status events. |
| `operator_prompt` | Dashboard prompt/manual confirmation, marked as guided. |

Do not block the database/history implementation on live camera capture. `capture_image` accepts an authored `image_path` or `DOGOPS_GO2_CAMERA_IMAGE_PATH` for configured Go2 frame capture, falls back to the placeholder demo image when no frame file exists, and stores metadata identifying the evidence source.

## Dashboard Changes

Add a route run panel near the existing route controls:

- Current route run state.
- Active waypoint/action.
- Progress: `waypoints reached / total`, `actions completed / total`.
- Last event and last error.
- Buttons: `Run Route`, `Stop Route`, later `Pause/Resume` if needed.
- Event timeline for the active route run.
- Evidence chips for images, tag scans, QR scans, and inspections.

- History table across all local DogOps runs with:

  - DogOps run ID,
  - route ID,
  - state,
  - duration,
  - waypoint count,
  - action count,
  - failures,

Add JSON endpoints:

## Migration Strategy

1. If `.dogops/dogops.sqlite` does not exist, create it lazily.
2. If `route_execution.json` exists and no route runs exist, import it as a legacy route run with best-effort event rows.
3. Continue writing `route_execution.json` until all dashboard/report code reads SQLite.
4. Keep JSONL exports forever as a debugging/export surface.

## Safety Rules

- `Stop Route` must remain available and should still attempt the local hard stop before MCP route stop.
- Only one active route run per DogOps run by default.
- Refuse to start a second route run while one is `running`, unless the user explicitly chooses `force stop and start`.
- All guided/operator-confirmed steps must record `guided=true`.
- Action failures should obey `required`: required failures fail the route run; optional failures are recorded and continue.
- Never claim QR/image capture happened unless an evidence row exists. Placeholder image evidence must be labeled as demo-generated until real Go2 camera capture is wired.
- Do not automatically prune local history in the first implementation.

## Tests

Focused tests:

- Creates SQLite schema lazily.
- Starts a route run and records immutable selected route snapshot.
- Writes waypoint events in order.
- Repeated route runs produce distinct `route_run_id` values and do not overwrite history.
- Route history lists runs across multiple DogOps run directories.
- Compatibility `route_execution.json` still updates.
- Dashboard history endpoint lists latest first.
- Action handler records successful `scan_tags`.
- Action handler records QR scan results or an honest unavailable/failure state.
- Capture-image action writes placeholder demo evidence before real camera support exists.
- Required action failure fails route run.
- Optional action failure records event and continues.
- Unified timeline includes incident, work-order, and verification events.
- Report includes route run summary.

## Implementation Phases

### Phase 1: Durable Route Runs

- Add global SQLite-backed `RouteRunStore`.
- Create one route-run record per `Run Route`.
- Persist route events to SQLite and `route_runs/<id>/events.jsonl`.
- Keep `route_execution.json` compatibility.
- Add tests for repeated route runs and current status.

### Phase 2: Dashboard History

- Add route-run list/current/detail endpoints.
- Add dashboard active timeline and history table.
- Include all timeline event types in the dashboard table.
- Link route-run events to map waypoints where practical.

### Phase 3: Planned Actions

- Add route action schema.
- Add action handlers for `scan_tags`, `scan_qr`, `inspect_asset`, `verify_work_order`, and `operator_prompt`.
- Record action events and observations.

### Phase 4: Evidence Capture

- Add evidence table and file layout.
- Add placeholder demo image capture and QR unavailable/failure states.
- Wire real Go2 image capture after the live camera stream path is validated.

### Phase 5: Reporting and Export

- Include route-run summaries in `report.json` and `report.md`.
- Export selected route run as JSON/JSONL bundle.
- Add comparison helpers for repeated route runs if needed.

## Remaining Hardware Follow-Ups

1. Validate the route-run history path in the foreground full-DimOS runtime against the real Go2.
2. Wire the live Go2 camera stream or frame-export path into the configured `image_path`/`DOGOPS_GO2_CAMERA_IMAGE_PATH` capture hook.
